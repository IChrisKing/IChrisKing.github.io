cjdns之干扰包设计

cjdnscjdns源码分析
  • C/C++
  • cjdns
  • cjdns源码分析

需求

在和peer建立连接之前，先发送一些干扰包来将自己伪装成其他协议。以此来欺骗防火墙的流量分析。

总体设计要点

  • 干扰包的使用时机：CA握手前互相发送
  • 干扰包的格式：选取建立在UDP协议基础上的应用层协议，比如l2tp
  • 干扰包与正常包的区分：干扰包头部标志位
  • 收到干扰包后的处理：分辨干扰包类型，根据类型构建同协议的新干扰包并返回。丢弃收到的干扰包。
  • 停止发送干扰包的契机：发包数量达到NOISE_PKG_ENOUGH，或者发包时间达到TIME_TO_SEND_NOISE_MILLISECONDS

设计模块

  • 干扰包结构体
  • 干扰包生成器
  • 干扰包分析方法
  • 干扰包发送方法
  • 收包处插入干扰包分辨方法
  • 发包处插入干扰包发送方法
  • 停止发干扰包的几个契机

以下根据程序中与peer建立连接的过程为引线，来分析整个干扰包运作的过程

第一个干扰包发送过程分析

当引擎启动后，会根据conf里面的配置，主动向inbound发送消息，这是第一个peer的建立，是主动连出的。
这里需要做的是，在发送真实的握手包数据前，先发出一些干扰包。这里引出的设计模块包括，

  1. peer中的成员struct NoisePackage_Peer* noisePkg;这是一个干扰包结构体
  2. InterfaceController的成员struct NoisePackage_Maker* noisePkgMk;这是一个干扰包生成器结构体
  3. 发包处插入的干扰包发送方法

干扰包结构体

干扰包主要包括三个成员

1. type

标识当前使用的干扰包的协议类型
包括

2. header

构造干扰包头部所必须的字段。
只是必须字段，不是全部字段。在一次会话中，需要前后稳定或前后联系的字段，都需要放在header中进行保存。比如sessionId这一类需要保持不变的字段。或者nr这一类需要依次递增的字段。
但如果header中有类似time这样的字段，只需要在构造包的时候临时获取即可。
以L2TP为例

3. pkgNum

干扰包计数器
记录已经发送和接收的干扰包的总数
同时，也是一个标志位，来决定是需要继续发送干扰包，还是可以开始发送正常数据包。

  1. 当发送和接收到干扰包时，pkgNum都会增加1，当达到一个预设数值NOISE_PKG_ENOUGH时，不再发送干扰包，开始发送正常数据包
  2. 当发送干扰包的时间达到5秒时，直接将pkgNum设为NOISE_PKG_ENOUGH，并开始发送正常数据包

干扰包生成器

干扰包生成器的成员包括

  1. eventBase 用来获取时间
  2. rand 用来生成随机数，这些随机数会用在构造干扰包头部某些随机字段
  3. alloc 分配干扰包内存

停止发送干扰包的契机

上面已经分析了NoisePackage_Peer和NoisePackage_Maker的结构，分析了干扰包构造，分析，发送的方法，分析了收包和发包处插入干扰包相关方法的位置和方式。
现在还剩下最后一个问题：什么时候停止发送干扰包
根据设计，停止发送干扰包的契机有两个

  1. 干扰包收发数量达到NOISE_PKG_ENOUGH
  2. 干扰包发送时间达到TIME_TO_SEND_NOISE_MILLISECONDS

是否要继续发送干扰包的判断在iciPing方法当中，iciPing的作用是，当引擎启动后，会以某个固定频率向peer发出ping操作，以此来保证自己和peer之间的联系没有中断。

所以，ep->timeOfLastMessage + ic->pingAfterMilliseconds + TIME_TO_SEND_NOISE_MILLISECONDS
近似于ep创建时间+TIME_TO_SEND_NOISE_MILLISECONDS
而ep创建时间与首个干扰包发送时间接近，所以，ep->timeOfLastMessage + ic->pingAfterMilliseconds + TIME_TO_SEND_NOISE_MILLISECONDS近似于首个干扰包发送时间+TIME_TO_SEND_NOISE_MILLISECONDS。
这就是第二个条件控制是否发送干扰包的方式。
